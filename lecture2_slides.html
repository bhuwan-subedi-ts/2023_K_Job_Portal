Relational Database Concepts

üóÉÔ∏è Table

Collection of related data organized in rows and columns

Example: Users table with user information

üìä Row (Record)

Single entry in a table containing all data for one item

Example: One user's complete information

üìã Column (Field)

Specific attribute or property of the data

Example: name, email, password columns

üîë Primary Key

Unique identifier for each row in a table

Example: user_id (1, 2, 3, 4...)

üîó Foreign Key

Reference to primary key in another table

Example: job application references user_id

üìê Relationships

How tables are connected to each other

Types: One-to-One, One-to-Many, Many-to-Many

üéØ Real-World Analogy: Library System

  • Table = Card Catalog: Organized information storage
  • Row = Individual Card: One book's information
  • Column = Category: Title, Author, ISBN
  • Primary Key = ISBN: Unique book identifier
  • Foreign Key = Author ID: Links to Authors table

Understanding Database Relationships

1Ô∏è‚É£ One-to-One (1:1)

Definition: Each record in Table A relates to exactly one record in Table B

Example: User ‚Üî User Profile

1Ô∏è‚É£ One-to-Many (1:N)

Definition: Each record in Table A can relate to multiple records in Table B

üîÑ Many-to-Many (N:M)

Definition: Multiple records in Table A relate to multiple records in Table B

SQL Basics - The Language of Databases

üìù What is SQL?

  • Structured Quary Language
  • Standard language for relational databases
  • Used to create, read, update, delete data
  • Declarative language (what, not how)

üéØ Main SQL Categories

DDL - Data Definition Language

Structure operations: CREATE, ALTER, DROP

DML - Data Manipulation Language

Data operations: INSERT, UPDATE, DELETE

DQL - Data Query Language

Data retrieval: SELECT

üîß Basic SQL Commands

-- Create a table CREATE TABLE users ( id SERIAL PRIMARY KEY, email VARCHAR(255) NOT NULL ); -- Insert data INSERT INTO users (email) VALUES ('john@example.com'); -- Select data SELECT * FROM users; -- Update data UPDATE users SET email = 'newemail@example.com' WHERE id = 1; -- Delete data DELETE FROM users WHERE id = 1;
‚ö†Ô∏è Important: Always use WHERE clause with UPDATE and DELETE to avoid modifying all records!

Creating the Users Table

üë§ Users Table Design

Fields and Data Types:

id: SERIAL PRIMARY KEY
Auto-incrementing unique identifier

email: VARCHAR(255) UNIQUE NOT NULL
User's email address (login credential)

password: VARCHAR(255) NOT NULL
Encrypted password hash

full_name: VARCHAR(255) NOT NULL
User's display name

user_type: VARCHAR(20) CHECK(...)
Either 'admin' or 'jobseeker'

created_at: TIMESTAMP DEFAULT CURRENT_TIMESTAMP
Account creation timestamp

üîß SQL Implementation

CREATE TABLE users ( id SERIAL PRIMARY KEY, email VARCHAR(255) UNIQUE NOT NULL, password VARCHAR(255) NOT NULL, full_name VARCHAR(255) NOT NULL, user_type VARCHAR(20) CHECK (user_type IN ('admin', 'jobseeker')) NOT NULL, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

üéØ Key Features Explained

  • SERIAL: Auto-incrementing integer
  • UNIQUE: No duplicate emails allowed
  • CHECK: Validates user_type values
  • DEFAULT: Automatically sets timestamp
  • NOT NULL: Field must have a value
‚ö†Ô∏è Security Note: Never store plain text passwords! We'll use bcrypt hashing in Node.js.

Creating the Jobs Table

CREATE TABLE jobs ( id SERIAL PRIMARY KEY, title VARCHAR(255) NOT NULL, description TEXT NOT NULL, company VARCHAR(255) NOT NULL, location VARCHAR(255), salary_range VARCHAR(100), posted_by INTEGER REFERENCES users(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

üîó Foreign Key Relationship

REFERENCES users(id)

  • Purpose: Links job to the admin who posted it
  • Constraint: posted_by must exist in users table
  • ON DELETE CASCADE: If user is deleted, their jobs are too

üìä Data Type Choices

  • TEXT: For long descriptions (no length limit)
  • VARCHAR(255): For shorter text fields
  • INTEGER: For foreign key references

üí° Design Decisions Explained

Why VARCHAR for salary_range?

Flexible format: "‚Çπ50,000 - ‚Çπ80,000", "Negotiable", "Up to ‚Çπ1,00,000"

Why location is nullable?

Some jobs might be "Remote" or location TBD

Why ON DELETE CASCADE?

If admin account is removed, their job postings should be removed too

‚ö†Ô∏è Alternative: You could use ON DELETE SET NULL to keep jobs but mark posted_by as NULL

Creating the Applications Table

CREATE TABLE applications ( id SERIAL PRIMARY KEY, job_id INTEGER REFERENCES jobs(id) ON DELETE CASCADE, applicant_id INTEGER REFERENCES users(id) ON DELETE CASCADE, cover_letter TEXT, status VARCHAR(20) DEFAULT 'pending' CHECK (status IN ('pending', 'reviewed', 'accepted', 'rejected')), applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, -- Prevent duplicate applications UNIQUE(job_id, applicant_id) );

üîó Multiple Foreign Keys

job_id ‚Üí jobs(id)

Which job was applied to

applicant_id ‚Üí users(id)

Which user applied

üéØ Junction Table Pattern

Applications table serves as a junction between Users and Jobs, enabling many-to-many relationship with additional data

‚ú® Advanced Features

UNIQUE(job_id, applicant_id)

Composite Unique Constraint: Prevents user from applying to same job twice

Status Workflow

  • pending: Just applied
  • reviewed: Admin has seen it
  • accepted: Candidate selected
  • rejected: Not selected
ü§î Question: What happens if a job seeker tries to apply for the same job twice?
Answer: PostgreSQL will return a unique constraint violation error!

Querying Our Data

üîç Basic Queries

-- View all users SELECT * FROM users; -- View only job seekers SELECT full_name, email FROM users WHERE user_type = 'jobseeker'; -- View all jobs SELECT title, company, location FROM jobs; -- Count applications per job SELECT job_id, COUNT(*) FROM applications GROUP BY job_id;

üîó JOIN Queries

-- Applications with job and user details SELECT u.full_name AS applicant_name, j.title AS job_title, j.company, a.status, a.applied_at FROM applications a JOIN users u ON a.applicant_id = u.id JOIN jobs j ON a.job_id = j.id; -- Jobs with poster details SELECT j.title, j.company, u.full_name AS posted_by_name, j.created_at FROM jobs j JOIN users u ON j.posted_by = u.id;

üéØ Understanding JOINs

  • INNER JOIN: Returns only matching records from both tables
  • LEFT JOIN: Returns all records from left table, matching from right
  • Foreign keys make JOINs possible - they define relationships

üí° Real-World Query Examples

Business Question: "Show me all pending applications for jobs posted this month"

SELECT u.full_name, j.title, a.applied_at FROM applications a JOIN users u ON a.applicant_id = u.id JOIN jobs j ON a.job_id = j.id WHERE a.status = 'pending' AND j.created_at >= DATE_TRUNC('month', CURRENT_DATE);
