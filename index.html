üîÑ How It Works:

  3. Server Processing: Handles business logic
  4. Database Query: Retrieves/stores data
  5. HTTP Response: Sent back to client

  6. Frontend shows confirmation

HTTP Protocol & REST APIs

üåê HTTP (HyperText Transfer Protocol)

Communication protocol between client and server

Common HTTP Methods:

  • GET: Retrieve data
  • POST: Create new data
  • PUT: Update existing data
  • DELETE: Remove data

HTTP Status Codes:

  • 200: Success
  • 201: Created
  • 400: Bad Request
  • 401: Unauthorized
  • 404: Not Found
  • 500: Server Error

üîÑ REST API Principles

REpresentational State Transfer

Key Principles:

  • Stateless: Each request is independent
  • Resource-based: URLs represent resources
  • HTTP Methods: Actions on resources
  • JSON Format: Data exchange format

Job Portal API Examples:

GET /api/jobs // Get all jobs POST /api/jobs // Create new job GET /api/jobs/123 // Get specific job PUT /api/jobs/123 // Update job DELETE /api/jobs/123 // Delete job POST /api/jobs/123/apply // Apply for job

Database Design Principles

üìä Database Design Process

  1. Requirements Analysis: What data do we need?
  2. Entity Identification: Main objects in our system
  3. Relationship Mapping: How entities connect
  4. Normalization: Reduce data redundancy
  5. Implementation: Create actual tables

üîë Key Concepts

  • Primary Key: Unique identifier
  • Foreign Key: Reference to another table
  • Relationships: One-to-One, One-to-Many, Many-to-Many
  • Constraints: Data validation rules

üóÑÔ∏è Our Job Portal Entities

üë§ Users

id, email, password, name, user_type, created_at

üíº Jobs

id, title, description, company, location, salary, posted_by, created_at

üìù Applications

id, job_id, applicant_id, cover_letter, status, applied_at

üîó Relationships

  • Users ‚Üí Jobs: One-to-Many (Admin posts jobs)
  • Users ‚Üí Applications: One-to-Many (User applies to jobs)
  • Jobs ‚Üí Applications: One-to-Many (Job has many applications)
